# 天地图 UI 设计技能

本技能使用接近 Anthropic `frontend-design` 的工作方式，并针对当前项目做小幅适配。

保留 `tianditu-ui-design` 这个包名，只是为了兼容当前项目的 package 路由与引用；真正的设计方法论以“先定明确审美方向，再生成高完成度界面”为核心。

目标不是单纯把颜色换好看，而是让地图页面具备鲜明但可信的设计方向，避免常见的通用 AI 风格、套模板感和视觉犹豫。

## 何时使用

- 用户明确要求页面更美观、更高级、更有质感
- 需要改首页、工作区、搜索结果页、信息面板、图层面板、卡片区
- 需要减少“AI 味”、避免普通后台模板感
- 需要补齐 loading / empty / error / hover / focus 等界面状态

## Design Thinking Before Coding

先定方向，再写代码。不要一上来直接拼组件。

开始实现前，先回答这几个问题：

1. 这个页面最重要的任务是什么
2. 地图是主舞台还是信息面板是主舞台
3. 页面应该呈现什么气质
4. 用户看完后最容易记住的视觉特征是什么

审美方向必须明确，不要停留在“现代、简洁、蓝色系”这种空话上。先选一个真正清晰的路线，再执行到位：

- 专业 GIS 控制台
- 编辑化信息面板
- 冷静政务风
- 数据新闻风
- 深色监控台
- 轻量研究工具风
- 工业感 / 工具感
- 柔和但克制的信息产品风

重点不是风格有多夸张，而是是否一致、是否有记忆点。

再补一层实现约束：

- 是否需要专业可信感，而不是营销感
- 是否需要高信息密度，而不是只做氛围
- 是否有桌面端 / 移动端 / 工作区布局限制
- 哪个区域必须最先被看见，哪个区域只做辅助

## Frontend Aesthetics Guidelines

1. 地图优先
地图永远是主舞台。视觉层级不得压制地图主信息，也不要让装饰元素喧宾夺主。

2. 不要做成通用后台模板
避免“白底卡片 + 蓝按钮 + 默认阴影 + 普通表单”的机械组合。

3. 不要落回常见 AI 审美
避免过度依赖紫色渐变、全圆角胶囊卡片、过于均匀的留白、机械分栏和千篇一律的 Inter 风。不要做 cookie-cutter 页面。

4. 一次只做一个明确方向
宁可做一个完成度高的方向，也不要把极简、科技、政务、营销风混在一起。

5. 真实状态必须完整
所有关键区域至少考虑这些状态中的大部分：
- `loading`
- `ready`
- `empty`
- `error`
- `hover`
- `active`
- `focus`
- `disabled`

6. 视觉变量集中管理
颜色、阴影、圆角、间距、字号统一抽成 CSS Variables，不允许散落全文件。

## 设计执行重点

### Typography

- 字体要有明确角色分工：标题、正文、数据、标签各自承担不同层级
- 避免默认字体堆导致页面没有性格
- 不要默认退回 `Inter / Roboto / Arial / system-ui`
- 信息密度高时优先保证可读性，再谈个性

### Color

- 先定主色调和中性色结构，再补强调色
- 地图类页面更适合“少量强强调 + 大面积克制背景”，而不是到处抢眼
- 状态色必须和业务语义一致，不要只追求漂亮
- 色彩应该服务任务，而不是“为了显得现代”

### Layout

- 鼓励做有主次关系的非平均分栏
- 允许不对称布局，但信息结构必须清楚
- 桌面端和移动端都要成立，不能只顾大屏截图
- 可以打破普通卡片栅格，但不能牺牲理解成本

### Motion

- 动效服务于层级感和反馈，不要只是“哪里都动一下”
- 优先做高价值时刻：首屏进入、卡片显现、面板展开、悬停反馈、列表切换
- 少量有记忆点的动效，优于到处撒微动效

### Detail

- 边框、阴影、分隔线、背景纹理、图标、角标都要统一语言
- 微差异很重要：对齐、密度、行高、留白、按钮反馈会决定页面是否粗糙
- 背景不必总是纯色，可以使用轻纹理、网格、渐变层或低调图形建立氛围

## 地图场景特别约束

1. 搜索页、规划页、工作区页面必须先明确：
- 地图区
- 控制区
- 结果区
- 详情区

2. 侧栏不能只是表单堆叠，要有明确的信息节奏：
- 任务入口
- 当前状态
- 结果反馈
- 后续操作

3. 地图上的交互反馈要和面板联动：
- 点选
- 高亮
- hover
- 当前项定位
- 结果筛选

4. 页面要保留专业可信感，特别是政务、地理信息、路线规划、灾害专题等场景，不要做成营销落地页。

## 工作流

1. 先输出简短设计简报
2. 再确定布局主次与视觉方向
3. 再补齐状态与交互细节
4. 最后输出完整可运行代码

## 输出协议

在写代码前，先输出一个简短设计简报，不要直接开写。至少包含：

- 页面目标
- 主任务与主舞台
- 审美方向
- 最重要的视觉记忆点
- 区域划分
- 关键状态
- 移动端策略

然后再输出代码。

## 输出要求

- 代码必须是可运行的前端实现，不要只给设计描述
- 优先做高完成度页面，不要回落到“普通后台 + 地图 iframe”式结构
- 如果风格选择偏极简，重点放在比例、层级、留白、字体和边界处理上
- 如果风格选择偏强表现力，重点放在主题一致性，不要杂乱堆料

## 最终检查

- 页面是否有明确审美方向，而不是“默认现代化”
- 地图是否仍然是主舞台
- 是否避免了明显的通用 AI 风格
- 是否补齐关键状态
- 是否使用 CSS Variables 管理视觉 token
- 是否兼顾桌面端和移动端
- 是否让用户一眼看得出主任务和下一步操作
