## 支持的数据类型

<Tabs>
  <TabItem label="GeoJSON">
    适合点、线、面地理数据。最常见于行政区划、边界专题、点位分布和路线展示。
  </TabItem>
  <TabItem label="JSON">
    适合结构清晰的业务数据，只要能提供坐标或与地图要素关联的字段即可。
  </TabItem>
  <TabItem label="Excel">
    适合业务人员手头最常见的数据来源。建议一行对应一个对象或地点，字段名尽量清楚。
  </TabItem>
</Tabs>

## 推荐节奏

<Steps>

1. 确定数据来源。

   先明确你是已经有 GeoJSON / JSON / Excel，还是需要先整理出一份结构化数据。

2. 准备最小可用数据。

   至少保证有名称、经纬度或边界、分类或说明字段。先不要追求一次把所有字段都准备齐。

3. 上传数据并说明目标。

   不要只说“帮我可视化”，而要补充“按省市分组”“地图和侧栏联动”“支持点击查看详情”等目标。

4. 先确认数据是否被正确读取。

   如果地图上没有出现点、面、路线或图层，先让系统检查数据结构，而不是继续要求美化。

5. 数据正确后再补交互和样式。

   在确认数据可用后，再逐步要求筛选、图例、统计卡片、弹窗和列表联动等增强能力。

6. 借助修复与巡检完成收口。

   如果页面报错或视觉不稳定，优先让系统做最小修复，而不是重做整页。

7. 页面稳定后再发布。

   当数据表达和页面结构都稳定后，再去发布分享链接并设置公开状态。

</Steps>

## 最小可用 JSON 示例

```json
[
  {
    "name": "示例点位 A",
    "category": "示例分类",
    "lng": 116.397,
    "lat": 39.907,
    "summary": "这里放业务说明或故事摘要"
  },
  {
    "name": "示例点位 B",
    "category": "示例分类",
    "lng": 121.4737,
    "lat": 31.2304,
    "summary": "这里放第二个点位的说明"
  }
]
```

## 数据路线里最容易忽视的事

<CardGrid>
  <Card title="先验证读取成功" icon="check-circle">
    如果数据还没被读对，继续叠加交互只会让问题更复杂。
  </Card>
  <Card title="字段名尽量直白" icon="list">
    名称、经纬度、分类、说明这些字段越直观，后续协作越顺畅。
  </Card>
  <Card title="先少量数据验证" icon="database">
    大批量数据建议先抽样验证一小份，跑通后再扩量。
  </Card>
  <Card title="大点位量优先图层方案" icon="seti:geojson">
    一开始就用大量 DOM 标注通常不稳定，更适合先做图层式表达。
  </Card>
</CardGrid>

<Aside type="caution" title="最容易踩坑的地方">
上传数据后如果地图空白，不要第一反应去改颜色和布局。先检查是否真的存在坐标、边界或路线，以及字段名是不是系统能理解的结构。
</Aside>
