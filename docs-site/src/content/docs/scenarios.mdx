如果用户第一次接触系统，只讲按钮和功能通常不够。更有效的方式，是先让他从场景里判断“这是不是我现在要解决的问题”。

## 常见场景分类

<CardGrid>
  <Card title="故事地图与专题叙事" icon="book">
    适合活动、历史、路线、保护专题、项目汇报等需要叙事节奏的页面。
  </Card>
  <Card title="点位专题与分布可视化" icon="seti:map">
    适合门店、机构、项目点位、服务资源分布和区域落点展示。
  </Card>
  <Card title="路线规划与出行说明" icon="route">
    适合活动动线、通勤引导、景区游览路线和交通说明页面。
  </Card>
  <Card title="行政区划与边界专题" icon="seti:geojson">
    适合辖区范围、规划片区、专题分色、区域统计和边界展示。
  </Card>
  <Card title="灾害、历史、活动专题" icon="warning">

  </Card>

## 每类场景需要准备什么

| 场景 | 适合准备什么 | 更推荐的路线 |
| --- | --- | --- |
| 故事地图 | 主题主线、章节结构、关键地点 | [故事地图路线](/docs/story-builder/) |
| 点位专题 | 点位名称、经纬度、分类、说明字段 | [数据驱动路线](/docs/data-builder/) |
| 路线专题 | 起终点、路线节点、图文说明 | 先故事，再补数据 |
| 行政区划专题 | 边界数据、区域字段、表达目标 | [数据驱动路线](/docs/data-builder/) |
| 灾害/历史专题 | 时间线、地点、图片、说明文案 | 先故事，再视情况补数据 |
| 已有第一版 | 明确哪些部分要改、哪些不要动 | [修改与质量保障](/docs/quality/) |

<Aside type="tip" title="对甲方更友好的表达方式">
场景页建议做成案例式卡片，而不只是列表。每张卡片可以固定放“适合做什么、需要准备什么、最终能产出什么”三类信息。
</Aside>

## 场景页里最值得补的内容

1. 每个场景配一张真实案例截图。
2. 每个场景给一条可直接复用的需求描述。
3. 对需要数据的场景，补一个“最低数据要求”小模块。
4. 对可分享展示的场景，补“最终交付形态”说明。
