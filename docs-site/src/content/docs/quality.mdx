真正的交付质量，往往不是“第一次生成得多好”，而是“你能不能稳定地把第一版收口到可交付状态”。这一页就是讲这个阶段该怎么做。

## 你会用到的四个能力

<CardGrid>
  <Card title="继续追问与增量修改" icon="refresh">
    把系统当成多轮协作助手，不满意的地方直接继续补充要求，而不是推翻重来。
  </Card>
  <Card title="自动修复" icon="warning">
    页面运行报错时，系统会结合当前代码和错误信息做最小改动修复。
  </Card>
  <Card title="视觉巡检" icon="eye">
    页面渲染完成后，系统会检查明显的空白、缺失和布局异常，用于辅助收口。
  </Card>
  <Card title="运行档案" icon="file">
    每一轮生成和修复都能被回看，适合培训、复盘和问题定位。
  </Card>
</CardGrid>

## 更稳妥的收口顺序

1. 先确认结构和地图表达正确。
2. 再修文案、层次和视觉细节。
3. 最后再检查分享页信息、公开方式和对外呈现。

<Aside type="tip" title="怎么看待自动修复和视觉巡检">
它们更像“帮你找问题、帮你先做一版最小修复”的助手，而不是绝对正确的裁判。正式交付前，仍然建议人工检查故事文案、统计口径和页面细节。
</Aside>

## 适合人工重点复查的内容

1. 标题、图例、说明文字是否和主题一致。
2. 数据口径、单位和统计说明是否准确。
3. 地图范围、层级和重点地点是否符合预期。
4. 分享页标题、描述和可见性是否与最终用途一致。
